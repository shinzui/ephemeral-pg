# Suite-level template databases

This guide describes a high-throughput fixture pattern for integration suites
that need a real PostgreSQL database per example, but also need to run an
expensive schema setup step such as SQL migrations.

The pattern is:

1. Start one cached PostgreSQL server for the whole suite with
   `EphemeralPg.startCached`.
2. Create one template database inside that server.
3. Run migrations or schema setup once against the template database.
4. For every example, create a fresh database with
   `CREATE DATABASE test_n TEMPLATE template_db`.
5. Run the example against the fresh database.
6. Drop the fresh database after the example finishes.

This keeps per-example isolation while avoiding repeated PostgreSQL server
startup and repeated migration work.

## When to use this pattern

Use this pattern when all of these are true:

- Your suite has many database-backed examples.
- Most examples need the same schema or mostly the same seed state.
- Schema setup is meaningfully expensive, for example because you run a
  migration tool, build indexes, install extensions, or create triggers.
- Tests need isolated data, so sharing one database across examples would hide
  ordering bugs or require manual cleanup.
- You are testing application behavior against PostgreSQL, not the behavior of
  PostgreSQL startup, shutdown, or `initdb`.

Good fits include event-store tests, queue adapter tests, ORM integration
suites, authorization and tenancy suites, and services that run migrations in a
deploy step before opening their runtime connection pool.

## When not to use it

Use plain `withCached` instead when:

- There are only a few database tests.
- Test time is dominated by application work rather than database setup.
- Each example needs a different PostgreSQL server configuration.
- The tests intentionally exercise server lifecycle behavior such as startup,
  restart, crash recovery, or configuration reloads.
- The extra fixture machinery would be more complex than the time it saves.

Use filesystem snapshots instead when you need to restore cluster-level state
outside one database, or when you need to test data-directory behavior. Template
database cloning is a PostgreSQL database-level copy; it is not a whole-cluster
restore.

Use dump/restore when you need a portable artifact, want to inspect SQL output,
or need to move state between different PostgreSQL servers. Dump/restore is
usually slower than template database cloning for per-example isolation.

## Why this is faster

`withCached` speeds up PostgreSQL startup by caching the result of `initdb`.
That is useful, but every `withCached` call still starts a PostgreSQL process,
creates a database, and then your test fixture usually runs migrations.

Template database cloning changes the shape of the cost:

- PostgreSQL process startup happens once per suite.
- Migrations happen once per suite.
- Each example pays only for `CREATE DATABASE ... TEMPLATE ...`, opening the
  application connections, and `DROP DATABASE`.

On suites where migrations dominate fixture setup, this can be a large
improvement while preserving the important property that every example receives
an empty, isolated database.

## Important PostgreSQL constraints

PostgreSQL only allows cloning from a template database when no other session is
connected to that template database. This is the main rule to design around.

Practical consequences:

- Run migrations through short-lived connections or pools.
- Release the migration pool before the first clone.
- Do not keep an application connection pool open against the template database.
- Run tests against cloned databases, not against the template.

PostgreSQL also does not parameterize database identifiers in `CREATE DATABASE`
or `DROP DATABASE`, so quote generated database names as identifiers. Keep names
simple and generated by your fixture.

`DROP DATABASE ... WITH (FORCE)` is available on modern PostgreSQL releases and
is convenient for test cleanup because it terminates remaining sessions. If you
support older PostgreSQL versions, terminate sessions from `pg_stat_activity`
before dropping the database.

## Complete fixture

The fixture below is deliberately ordinary Haskell. It does not require a test
framework-specific API; use it from `hspec`, `tasty`, or a custom harness.

The suite fixture returns a `Fixture` that owns one running PostgreSQL server and
one migrated template database. `withFreshDatabase` clones that template and
passes a libpq connection string for the clone to the example.

```haskell
{-# LANGUAGE BlockArguments #-}
{-# LANGUAGE OverloadedRecordDot #-}
{-# LANGUAGE OverloadedStrings #-}

module Test.Postgres (
  Fixture,
  withMigratedSuite,
  withFreshDatabase,
) where

import Control.Concurrent.STM (TVar, atomically, newTVarIO, stateTVar)
import Control.Exception (bracket, onException)
import Data.Text (Text)
import Data.Text qualified as Text
import EphemeralPg qualified as Pg
import Hasql.Connection.Settings qualified as Conn
import Hasql.Pool qualified as Pool
import Hasql.Pool.Config qualified as Pool.Config
import Hasql.Session qualified as Session

data Fixture = Fixture
  { server :: Pg.Database
  , templateName :: Text
  , nextId :: TVar Int
  }

withMigratedSuite :: (Fixture -> IO a) -> IO a
withMigratedSuite action = do
  started <- Pg.startCached Pg.defaultConfig Pg.defaultCacheConfig
  case started of
    Left err -> fail (Text.unpack (Pg.renderStartError err))
    Right server ->
      bracket
        (setup server `onException` Pg.stop server)
        stopFixture
        action
 where
  setup server = do
    counter <- newTVarIO 0
    let fixture = Fixture server "app_template" counter
    runSql server "CREATE DATABASE app_template"
    migrate (connectionStringFor server "app_template")
    pure fixture

  stopFixture fixture =
    Pg.stop fixture.server

withFreshDatabase :: Fixture -> (Text -> IO a) -> IO a
withFreshDatabase fixture action =
  bracket create dropDb \dbName ->
    action (connectionStringFor fixture.server dbName)
 where
  create = do
    n <- atomically $ stateTVar fixture.nextId \i -> (i + 1, i + 1)
    let dbName = "test_" <> Text.pack (show n)
    runSql fixture.server $
      "CREATE DATABASE " <> quoteIdentifier dbName
        <> " TEMPLATE " <> quoteIdentifier fixture.templateName
    pure dbName

  dropDb dbName =
    runSql fixture.server $
      "DROP DATABASE IF EXISTS " <> quoteIdentifier dbName <> " WITH (FORCE)"

migrate :: Text -> IO ()
migrate connStr =
  runSqlOn connStr "CREATE TABLE events (id bigserial PRIMARY KEY)"

runSql :: Pg.Database -> Text -> IO ()
runSql db = runSqlOn (Pg.connectionString db)

runSqlOn :: Text -> Text -> IO ()
runSqlOn connStr sql = do
  pool <- Pool.acquire $
    Pool.Config.settings
      [ Pool.Config.staticConnectionSettings (Conn.connectionString connStr)
      , Pool.Config.size 1
      ]
  result <- Pool.use pool (Session.script sql)
  Pool.release pool
  either (fail . show) pure result

connectionStringFor :: Pg.Database -> Text -> Text
connectionStringFor db dbName =
  Text.unwords
    [ "host=" <> Text.pack db.socketDirectory
    , "port=" <> Text.pack (show db.port)
    , "dbname=" <> dbName
    , "user=" <> db.user
    ]

quoteIdentifier :: Text -> Text
quoteIdentifier ident =
  "\"" <> Text.replace "\"" "\"\"" ident <> "\""
```

Replace `migrate` with your real migration runner. For example, an application
that uses a migration library can call it there, then release every migration
connection before returning from `setup`.

## Hspec integration

One simple `hspec` shape is to start the suite fixture in `main` and let
individual examples clone databases:

```haskell
{-# LANGUAGE BlockArguments #-}

import Test.Hspec
import Test.Postgres

main :: IO ()
main =
  withMigratedSuite \fixture ->
    hspec do
      describe "repository" do
        it "inserts and reads rows" do
          withFreshDatabase fixture \connStr -> do
            -- Open your application pool with connStr.
            -- Run the example.
            pure ()
```

This keeps the fixture independent of `hspec` internals. If you prefer `around`
or `aroundAll`, keep the same lifecycle boundaries: start the server and migrate
the template once, then clone databases around individual examples.

## Tasty integration

With `tasty`, start the suite fixture around `defaultMain` and call
`withFreshDatabase` inside each test:

```haskell
{-# LANGUAGE BlockArguments #-}

import Test.Tasty
import Test.Tasty.HUnit
import Test.Postgres

main :: IO ()
main =
  withMigratedSuite \fixture ->
    defaultMain $
      testGroup
        "repository"
        [ testCase "inserts and reads rows" do
            withFreshDatabase fixture \connStr -> do
              -- Open your application pool with connStr.
              -- Run the assertion.
              pure ()
        ]
```

For very large `tasty` suites, you can also use `withResource` to thread the
fixture through the tree. The same rule applies: clone databases per test, not
per group, when you need data isolation.

## Parallel tests

This pattern can work with parallel tests, but there are two things to keep in
mind.

First, database names must be unique. Use an atomic counter, UUID, or another
process-local unique suffix. The example uses an STM `TVar`.

Second, many tests cloning and dropping databases at the same time can create
short bursts of PostgreSQL catalog work. If you see contention in very parallel
suites, limit database-heavy tests to a smaller parallelism level or use one
suite fixture per test group.

The template database itself must remain idle. Parallel examples should only
connect to their cloned databases.

## Seeding data

If every test needs the same seed data, insert that data into the template
database after migrations and before cloning starts. The clone will include that
seed data.

Good seed data:

- static lookup tables;
- extension setup;
- reference users or tenants used by most tests;
- expensive indexes or materialized views that are part of baseline state.

Avoid seed data that embeds timestamps, random values, or mutable counters unless
every example really should start from that exact value. When in doubt, put only
schema in the template and let each example create its own test data.

## Connection pools and cleanup

Application pools should be scoped inside `withFreshDatabase`. When the example
returns, close the application pool before `withFreshDatabase` drops the cloned
database.

If cleanup fails because a connection is still active:

1. Check that the application pool is closed before returning from the example.
2. Check that background workers, notification listeners, or subscription
   threads are stopped before cleanup.
3. Use `DROP DATABASE ... WITH (FORCE)` on PostgreSQL versions that support it.
4. On older PostgreSQL versions, terminate sessions from `pg_stat_activity`
   before `DROP DATABASE`.

## Choosing between mechanisms

| Mechanism | Best for | Isolation boundary | Typical cost |
|---|---|---|---|
| `with` | Simple tests and lifecycle coverage | One server per use | Full `initdb` + startup |
| `withCached` | Simple isolated DB tests | One server per use | Cached copy + startup |
| Template DB clones | Large suites with shared migrated schema | One database per example inside one server | Clone + app connection setup |
| Snapshots | Restoring data-directory or cluster-level state | Whole data directory | Snapshot/restore + restart |
| Dump/restore | Portable fixtures and inspectable SQL artifacts | Database contents | Usually slower than template clones |

Template database cloning is not a replacement for every fixture. It is a
targeted optimization for the common case where most examples need the same
schema and only their data needs to be isolated.
